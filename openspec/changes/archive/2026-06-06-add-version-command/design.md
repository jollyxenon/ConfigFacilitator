## Context

ConfigFacilitator 是一个 Go CLI 工具，目前使用手写的参数解析（无外部依赖）。CLI 支持多个子命令（new、sync、switch 等），并在 `internal/cli/cli.go` 中实现参数分发。当前没有版本显示功能，用户无法知道正在使用的版本。

## Goals / Non-Goals

**Goals:**
- 提供 `cfgfc --version` 和 `cfgfc -v` 命令来显示版本号
- 支持构建时版本注入，便于发布管理
- 版本信息简洁明了，仅显示版本号
- 与现有 CLI 参数解析无缝集成

**Non-Goals:**
- 不实现详细的版本信息（如构建时间、Git 提交等）
- 不添加版本比较或更新检查功能
- 不修改现有子命令的版本显示

## Decisions

### 1. 版本变量位置
**决策**: 在 `internal/cli/cli.go` 中定义包级变量 `var version = "dev"`
**理由**: 保持版本信息靠近 CLI 逻辑，便于维护。使用 "dev" 作为默认值，构建时通过 ldflags 注入实际版本。

**替代方案**:
- 在单独的 `version.go` 文件中定义：增加了文件数量，但可能更清晰
- 在 `cmd/cfgfc/main.go` 中定义：离入口点更近，但需要传递给 cli 包

### 2. 版本注入机制
**决策**: 使用 Go 的 `-ldflags "-X ..."` 在构建时注入版本号
**理由**: 这是 Go 社区的标准做法，无需外部依赖，支持 CI/CD 流程。

**替代方案**:
- 使用 `go:embed` 嵌入版本文件：需要额外的文件管理
- 使用环境变量：运行时依赖，不够可靠

### 3. 参数解析策略
**决策**: 在 `run()` 函数中添加版本检查，与现有帮助检查并行
**理由**: 保持现有解析逻辑的一致性，版本检查应在帮助检查之前，因为版本是独立功能。

**具体实现**:
- 添加 `isVersionArg()` 函数检查 `--version` 或 `-v`
- 在 `run()` 函数开始处检查版本参数
- 版本参数可以出现在任何位置（类似帮助参数）

### 4. 版本输出格式
**决策**: 仅输出版本号，无额外文本
**理由**: 简洁明了，便于脚本解析。格式：`1.0.0`

**替代方案**:
- 输出 `cfgfc version 1.0.0`：更详细，但可能冗余
- 输出带换行符的版本号：便于阅读，但可能影响脚本解析

## Risks / Trade-offs

**[风险] 版本注入失败** → 构建时使用默认 "dev" 版本，确保功能始终可用
**[风险] 参数冲突** → 使用 `--version` 和 `-v` 两个标志，减少冲突可能性
**[权衡] 版本信息简洁性** → 选择仅显示版本号，牺牲了详细信息换取简洁性
**[权衡] 参数位置灵活性** → 允许版本标志出现在任何位置，增加了灵活性但可能影响解析性能

## Migration Plan

1. 添加版本变量和解析逻辑
2. 更新构建配置以支持版本注入
3. 添加测试用例验证版本显示
4. 更新帮助信息显示版本标志
5. 验证现有功能不受影响

## Open Questions

- 是否需要支持 `-V`（大写 V）作为版本标志的别名？
- 版本标志是否应该像帮助标志一样在子命令中可用？（当前设计仅支持根命令）
